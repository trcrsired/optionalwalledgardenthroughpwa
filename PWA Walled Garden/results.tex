\section{Evaluation}

Chromium’s Rule of 2 states that no system should simultaneously combine untrusted input, unsafe implementation languages, and high privilege. We generalize this into a four-factor model suitable for operating systems and app ecosystems by adding a fourth dimension: patch latency. A system that is safe in theory but slow to update in practice remains vulnerable. We call this the \textbf{Rule of 4}:

\begin{enumerate}
    \item \textbf{Input trust} — whether the system accepts arbitrary, potentially malicious inputs.
    \item \textbf{Implementation safety} — whether applications are written in memory-safe languages.
    \item \textbf{Privilege level} — whether applications run with high or low system privilege.
    \item \textbf{Patch latency} — how quickly vulnerabilities can be fixed and deployed.
\end{enumerate}

A secure system must avoid the dangerous intersection of all four. We evaluate S/P Mode using this Rule of 4 and validate the model with four PWAs we implemented: the PWA Store\cite{pwastore}, PWA Player\cite{pwaplayer}, PWA Geolocation\cite{pwageolocation}, and WasmPass\cite{wasmpass}. All four applications are released as free and open source software under the GPLv3 license, ensuring full transparency, auditability, and user freedom. We then evaluate performance, usability, environmental impact, and compare S/P Mode to Android and iOS.

\subsection{Security Evaluation Using the Rule of 4}

\subsubsection{Input Trust}

S/P Mode removes the dominant source of untrusted input: arbitrary URLs, downloads, search results, and native binaries. All applications must be installed from the \textbf{PWA Store}, deployed on 2025-07-01. As of 2026-03-01, it contains over 1,000 PWAs. The store demonstrates:

\begin{itemize}
    \item \textbf{Decentralization}: deployed on Cloudflare Pages with no servers.
    \item \textbf{Zero infrastructure cost}: it stores only URLs, not binaries.
    \item \textbf{Transparency}: each PWA lists its origin URL for public inspection.
    \item \textbf{Safety}: each PWA is scanned using ScamAdvisor.
\end{itemize}

S/P Mode differs fundamentally from Apple’s App Store, Google Play, or Flatpak repositories, which host binaries, require heavy infrastructure, and obscure provenance. The PWA Store shows that a global app ecosystem can be maintained at near-zero cost while remaining transparent and auditable.

\subsubsection{Implementation Safety}

PWAs are written in HTML, CSS, and JavaScript—memory-safe languages. WebAssembly provides near-native performance while enforcing strict bounds checking and sandboxing.

To validate this, we implemented \textbf{WasmPass}, a WebAssembly port of the C++ fast\_io password generator. Generating 10 million passwords showed no noticeable slowdown compared to native C++, while all memory accesses were checked by the WebAssembly runtime. Combined with recent work on WebAssembly memory tagging~\cite{wasmMemoryTagging2024}, this demonstrates that CPU-intensive workloads can be executed safely without exposing the system to C/C++ memory-safety vulnerabilities.

\subsubsection{Privilege Level}

PWAs run inside the browser sandbox, and in S/P Mode the browser itself is sandboxed by the OS. PWAs cannot access arbitrary files, kernel interfaces, or native APIs. They only receive capabilities explicitly exposed by the browser.

We validated this by implementing PWAs that use battery status, screen lock, geolocation, WebUSB, and WebBluetooth. These experiments show that PWAs can interact with real hardware while remaining fully sandboxed. For example:

\begin{itemize}
    \item GrapheneOS can be flashed via WebUSB in Chromium browsers.
    \item An e-scooter can be controlled (e.g., beep) via WebBluetooth.
    \item Real-time device orientation, heading, speed, altitude, and high-accuracy GPS coordinates can be accessed through the Geolocation and DeviceOrientation APIs, as demonstrated by our GPLv3 PWA Geolocation application. It computes cardinal direction, multiple distance metrics, and reverse-geocoded place names entirely within the browser sandbox.
\end{itemize}

These capabilities require no native code and no elevated privilege.

\subsubsection{Patch Latency}

PWAs update automatically via service workers from their origin servers. The PWA Store is not the update channel; it only provides initial installation. This ensures:

\begin{itemize}
    \item zero user intervention,
    \item no OS-level installers,
    \item no reboot requirements,
    \item immediate rollout of security fixes.
\end{itemize}

Browsers also update frequently and independently of the OS. In contrast:

\begin{itemize}
    \item Android updates are delayed by OEMs and carriers,
    \item iOS updates are tied to device support lifecycles,
    \item both platforms lock bootloaders, preventing long-term maintenance.
\end{itemize}

S/P Mode does not restrict bootloaders or OS installation. Users can reinstall or replace the OS, run diagnostics (e.g., MemTest86), or install alternative systems such as Linux or FreeBSD. This extends device lifespan and reduces e-waste.

\subsection{Security Case Study: PWA Player vs.\ MPV}

To demonstrate the structural security benefits of PWAs, we compare our \textbf{PWA Player} with the native MPV media player. MPV is a high-quality open-source player, but like all native C/C++ applications, it has a history of memory-safety vulnerabilities, such as:

\begin{itemize}
    \item \textbf{CVE-2021-30145}: format string vulnerability in M3U playlist handling, enabling code execution.
    \item \textbf{CVE-2020-19824}: arbitrary code execution via the \texttt{ao\_c} parameter.
\end{itemize}

These vulnerabilities exist because MPV parses complex, attacker-controlled formats in an unsafe language at high privilege.

PWA Player, by design, cannot suffer from these classes of bugs:

\begin{itemize}
    \item it is written in JavaScript (no buffer overflows or format-string primitives),
    \item it delegates parsing and decoding to the browser’s hardened media pipeline,
    \item it runs inside the browser sandbox,
    \item it updates automatically via service workers.
\end{itemize}

The same crafted M3U file that could exploit MPV becomes harmless when passed to the browser’s media engine. Entire categories of MPV CVEs are structurally impossible in PWA Player.

There is also a size contrast: MPV’s Android APK (22.9 MB, SHA-256: b6926fa9...) bundles its own logic and decoders. PWA Player occupies only about 256 KB after installation on Android with Microsoft Edge, yet supports music, video, IPTV (M3U8), battery display, screenshots, and video recording—because these features are implemented by the browser engine, not by custom native code.

\subsection{Performance Evaluation}

Our three PWAs demonstrate that the PWA-based walled garden is not only safer but also performant:

\begin{itemize}
    \item \textbf{PWA Store}: installation is instant; no APK downloads or OS-level installers.
    \item \textbf{PWA Player}: uses the browser’s optimized media pipeline; we observed no perceptible playback degradation for IPTV or local media.
    \item \textbf{WasmPass}: shows that WebAssembly can handle tens of millions of operations with negligible slowdown relative to native C++.
\end{itemize}

Because PWAs share a single browser engine, they avoid the duplication and bloat of native frameworks and Electron-style apps. On Android, 54 installed PWAs plus browser cache consumed 3.06 GB total; the browser itself accounts for roughly 1 GB, leaving about 2 GB for all PWAs and their caches.

\subsection{Usability Evaluation}

S/P Mode is designed to provide a simple, predictable, and familiar experience while preserving full user autonomy. Its usability advantages arise from the S/P toggle and the separation between a locked-down environment (S/P Mode) and a fully open environment (P Mode). The key usability properties are:

\begin{enumerate}
    \item The S/P toggle appears directly on the login screen, giving users explicit control over whether they want a locked-down environment.
    \item S/P Mode runs only PWAs from the PWA Store. P Mode is the normal Windows environment capable of running any application.
    \item P Mode can install PWAs into S Mode, allowing power users to install PWAs directly from browsers.
    \item S/P Mode does not disable system services such as SSH servers or remote desktop.
    \item P Mode allows rebooting into UEFI, reinstalling other operating systems, and running diagnostic tools.
    \item S/P Mode can allow a limited set of signed native applications (e.g., Chrome, Firefox, WeChat, Android emulators, Microsoft Office) during a transition period.
\end{enumerate}

S/P Mode is universally applicable to GNU/Linux, macOS, and FreeBSD. It does not depend on Windows-specific features; Windows is used only as a concrete example.

\subsubsection{Comparison to iOS and Android}

From a usability perspective, S/P Mode offers several advantages over iOS and Android:

\begin{itemize}
    \item \textbf{Reversibility}: iOS and Android enforce a permanent walled garden. S/P Mode is optional and reversible.
    \item \textbf{Developer freedom}: iOS requires App Store approval; Android increasingly restricts sideloading. P Mode allows full developer workflows.
    \item \textbf{Device control}: iOS and Android prevent users from accessing bootloaders or reinstalling the OS. P Mode preserves full device control.
    \item \textbf{Application flexibility}: PWAs are cross-platform, lightweight, and automatically updated.
    \item \textbf{Transition path}: iOS and Android offer no migration path between locked-down and open environments. S/P Mode and P Mode coexist.
\end{itemize}

\subsection{Environmental Impact}

Because PWAs are small, memory-safe, and automatically updated, they extend device lifespan. S/P Mode does not lock bootloaders or prevent OS replacement, allowing users to maintain devices long after OEM support ends. In contrast, Android and iOS enforce planned obsolescence through locked bootloaders, proprietary firmware, and short update windows.

\subsection{Summary}

S/P Mode avoids the dangerous intersection of the Rule of 4. The PWA Store constrains inputs and demonstrates decentralized, zero-infrastructure distribution. PWA Player shows that entire classes of native CVEs are structurally impossible. WasmPass shows that CPU-intensive workloads can be executed safely and efficiently. S/P Mode is simpler, safer, more transparent, more repairable, and more environmentally sustainable than Android or iOS.

Finally, Windows is used only as a concrete example. The S/P Toggle is a general design pattern applicable to any non-walled-garden operating system. Once published, S/P Mode provides a technical and legal foundation for requiring platform vendors to support open, repairable, user-controlled computing environments.
